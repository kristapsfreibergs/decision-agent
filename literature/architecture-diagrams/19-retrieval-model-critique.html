Retrieval Model Critique — Top-k vs Evidence Authority Graph

Back to index

The core retrieval primitive used by virtually all current RAG systems — k-nearest neighbour over dense vectors — is the wrong primitive for governed enterprise decision systems. This is not a tuning problem. It is a mismatch between the objective function of the retrieval model and the requirements of auditable, complete-evidence decisions. This diagram makes that argument, then specifies what a better retrieval primitive looks like.

1 — The Failure Mode of Top-k

⚠ Top-k retrieval — what happens in practice

The compliance blacklist and ISO certificate are mandatory for a governed procurement decision but use legal/technical language that embeds far from the query. They land outside the k boundary and are silently excluded. The model never knows they exist.

✓ Evidence authority graph — class-first, score-second

DSC defines required classes. All documents in each required class are fetched regardless of embedding distance. PAAP ranks them by authority, not similarity.

vendor_proposal REQUIRED

Vendor B proposal 0.56

The compliance blacklist and ISO certificate are guaranteed present because compliance_rule is a required class. Their embedding distance is irrelevant. Missing a required class is a validation failure — the worker does not run.

2 — Query Model: What Each Database "Thinks" a Query Is

⚠ Vector DB — k-NN query model

-- Standard RAG retrieval query SELECT doc_id, content, cosine_similarity(embedding, $query_vec) AS score FROM documents ORDER BY score DESC LIMIT $k -- Assumption: score ≈ relevance -- No required classes -- No authority weighting -- No completeness check -- Silent failure if k+1 is mandatory

The entire model is a single parameter: how many documents to take. There is no concept of "this class of evidence is required." There is no concept of "this source is more authoritative than that one." There is no way to know what was not retrieved.

✓ Evidence Authority Graph — scope-first query model

-- Evidence Authority Graph retrieval SCOPE('procurement_v1') -- loads DSC required: [vendor_proposal, compliance_rule, signed_contract] optional: [market_benchmark] blocked: [personal_data] FETCH_REQUIRED(compliance_rule) -- ALL, no k cutoff FETCH_REQUIRED(vendor_proposal) FETCH_REQUIRED(signed_contract) SCORE(evidence_set) -- PAAP authority × temporal × corroboration VALIDATE(evidence_set, scope) -- error if required class empty RECALL_SIMILAR($query, optional_classes, k=5) -- vector search is a supplement, not the gate

Completeness is the primary gate. Vector similarity is a supplementary signal for optional classes only. The query cannot succeed if a required evidence class is empty — it raises a validation error that surfaces to the operator.

3 — Dimension-by-Dimension Comparison

Dimension Vector DB + top-k Evidence Authority Graph
Retrieval primitive Approximate nearest neighbour (ANN) in embedding space. Returns k most similar vectors. Scope-first class fetch + authority graph traversal. Returns all evidence within required scope, ranked by authority.
Completeness guarantee None. If the critical document is #(k+1), it is silently excluded. The model never knows. Hard guarantee for required classes. Empty required class = explicit validation failure, not silent omission.
Ranking signal Cosine similarity to query embedding. Rewards terminology match, penalises legal/technical language divergence. Authority weight × temporal decay × corroboration. Independent of query phrasing. A signed contract ranks above a summary regardless of wording.
Missing evidence signal None. The context window is silently shorter or padded with lower-quality results. Explicit: VALIDATE() raises a typed error — "compliance_rule evidence class is empty." Operator can act before the decision is made.
Data access control External to retrieval. Must be applied as a post-filter or injected into the query — easy to bypass or misconfigure. Built into SCOPE(). Blocked classes never enter the retrieval pipeline. DSC is enforced at the query seam, not by the model.
Audit trail Typically: what the model saw. Not why it was retrieved, what authority it had, or what was not retrieved. Every EvidenceSource has: id, class, authority_score, temporal_score, corroboration_count, why_included. Full replay possible from audit.jsonl.
Cross-run consistency Same query on updated index returns different results. Decisions are not reproducible. Evidence bundle locked at run creation time (snapshot). The same decision can be reproduced exactly from any date.
Suitable for Q&A, summarisation, coding assistants — tasks where partial context is acceptable and completeness is not a legal requirement. Procurement, compliance review, risk assessment, audit — tasks where missing evidence creates legal liability or wrong consequential actions.

4 — Specification: Evidence Authority Graph as a New Database Type

This is not a configuration of an existing database. It is a different data model with different query primitives, different ranking semantics, and different failure modes. The claim: AI agents making governed decisions need this primitive, and it does not exist yet as a first-class database product.

Data model

EvidenceNode
id, content, evidence_class, source_id, created_at, run_id (snapshot key)
AuthorityEdge
CITES | CORROBORATES | CONTRADICTS | SUPERSEDES — typed, directed, weighted
SourceNode
authority_weight (0.0–1.0 by tier), type, credential_ref (vault pointer, never inline)
ScopeContract
required_classes[], optional_classes[], blocked_classes[], min_authority_threshold

Query primitives

SCOPE(type)
Load DSC for this decision type. Sets required/optional/blocked class filters for the session.
FETCH_REQUIRED(class)
Return all EvidenceNodes of class within scope. No k cutoff. Errors if source is inaccessible.
SCORE(nodes)
Walk AuthorityEdges. Compute PAAP score per node: authority × temporal × corroboration × (1−conflict). Deterministic.
VALIDATE(set, scope)
Assert all required classes are non-empty and min_score met. Raises typed error on failure — not a soft warning.
RECALL_SIMILAR(q, classes, k)
Optional: vector ANN over optional_classes only. Supplements required evidence; never substitutes for it.
Why this is a new database type, not a new query against an existing one: Relational DBs (PostgreSQL) have no concept of evidence class, authority weight, or completeness validation. Vector DBs (Qdrant, pgvector) have no concept of required classes, scope contracts, or typed evidence relationships. Graph DBs (Neo4j) have no built-in authority propagation formula or evidence snapshot semantics. What is needed is a system where completeness, authority, and scope are first-class query constraints — not post-filters applied outside the database by application code that can be forgotten, misconfigured, or bypassed. The database enforces the governance contract at the query seam.

5 — Closest Existing Technologies and Their Gaps

pgvector + PostgreSQL

What it gives: vector search inside a relational DB. Row-level security. Full-text BM25. Transactional consistency.

What it lacks: no evidence class primitives, no authority propagation, no scope contract enforcement at query time. DSC must be enforced by application code — outside the DB.

Role in decision-agent: best V1 storage backend. Implement the Evidence Authority Graph semantics as a service layer on top.

Neo4j / Kuzu

What it gives: typed relationships, graph traversal (Cypher / Kùzu GQL), property graphs. Authority edges map naturally to graph edges.

What it lacks: no built-in PAAP scoring, no scope contract concept, no snapshot/run isolation. Vector search is secondary.

Role in decision-agent: best fit for the ontology layer (Vendor → Contract → Clause). PAAP scoring runs as a graph algorithm on top.

Weaviate / Qdrant

What it gives: vector + keyword hybrid search. Weaviate has object types and cross-references. Multi-tenancy.

What it lacks: no required-class guarantees, no authority weighting, no completeness validation. k is still the primary cutoff.

Role in decision-agent: semantic supplement for optional evidence classes where terminology variation is high. Not the primary retrieval primitive.

Thesis claim — stated precisely: Current RAG architectures use similarity as a proxy for relevance and k as a proxy for completeness. Neither proxy holds in governed enterprise decisions. The alternative is a retrieval model where completeness is enforced by a scope contract, relevance is defined by evidence class membership, and quality is ranked by authority propagation — not by distance to a query embedding. This architecture (DSC + PAAP + data gateway) is what Decision Agent implements, and it constitutes a distinct retrieval primitive from k-NN that is better suited to any domain where missing evidence creates legal, financial, or operational liability.

7 — The Research Frontier: What Comes After the Current Ceiling

Vector + graph + semantic retrieval is the current production ceiling. No published approach consistently beats this combination across enterprise retrieval tasks. But six research directions could change that — and this work sits at the intersection of several of them.

Neurosymbolic systems IBM, MIT — academic

Combine formal symbolic rules with learned neural representations. Formal rules give completeness and logical guarantees; neural components handle terminology variation and ambiguity.

Why it matters here: DSC + PAAP is already a working approximation of this. DSC is the symbolic layer — explicit, auditable, machine-checkable scope rules. Vector search is the neural layer. PAAP is the bridge — a deterministic scoring function that operates on neural-retrieved evidence using symbolic authority metadata.

No production-ready database product exists. The research gap this work fills: a deployed, working neurosymbolic retrieval system for enterprise decisions, not just an academic demo.

Probabilistic databases MayBMS, Trio — academic

Replace deterministic authority scores with probability distributions over facts. Instead of authority_weight = 0.95, a source node holds P(claim | source) = Beta(α, β). Uncertainty propagates through chains of evidence using Bayesian update rules.

Why it matters here: PAAP's deterministic formula is a pragmatic approximation of probabilistic reasoning. A probabilistic database would formalise what PAAP does manually — and allow confidence intervals on final evidence scores, not just point estimates.

Not production-ready. MayBMS runs on PostgreSQL. The framing for this work: PAAP is a deterministic approximation of probabilistic evidence scoring — a conscious engineering tradeoff for auditability and speed.

Causal graphs DoWhy, CausalNex — research

Move beyond correlation in evidence retrieval to causation. A vendor having low ISO scores causes compliance risk — it does not merely correlate with it. Causal graphs encode these directed relationships; retrieval follows causal paths, not similarity paths.

Why it matters here: Procurement risk assessment is inherently causal, not correlational. The risk_assessor worker reasons over causal chains (vendor quality → delivery risk → project failure), but the current evidence model is flat. A causal layer on top of the authority graph would allow workers to retrieve evidence along causal paths, not just class membership.

DoWhy (Microsoft) and CausalNex (QuantumBlack) are the closest production tools. A future extension: evidence edges typed as CAUSES, PREVENTS, CORRELATES_WITH rather than just CORROBORATES.

Learned sparse retrieval SPLADE, DeepCT — near-production

Learns sparse token-level representations instead of dense embeddings. Outperforms dense vectors on out-of-domain and technical queries because it preserves exact terminology. A compliance clause written in legal Latin does not drift to a distant embedding position — the sparse representation keeps the token weight high.

Why it matters here: The terminology mismatch failure (compliance blacklist ranking low against a business-language query) is directly addressed by learned sparse retrieval. This is the right technology for the vector supplement layer in V1 — better than BM25 alone, cheaper than late interaction.

SPLADE models are available on HuggingFace. Drop-in replacement for the BM25 layer in the retrieval stack. Addresses the same failure mode as this diagram's core argument, from the retrieval side rather than the completeness side.

ColBERT / late interaction Stanford — research to production

Token-level matching between query and document instead of single compressed vector comparison. Each query token attends to each document token independently — a much richer match signal. Particularly precise for multi-aspect queries where several distinct requirements must all be met.

Why it matters here: Procurement queries are multi-aspect: price, delivery, compliance, track record must all be satisfied. A single dense vector cannot represent four orthogonal requirements simultaneously. ColBERT-style scoring would give more precise recall within the optional evidence classes where vector search supplements required-class fetching.

RAGatouille library wraps ColBERT for production use. Higher storage and latency cost than dense vectors — justified only for high-stakes optional evidence recall, not for the required-class fetch (which is class-based, not similarity-based).

Temporal knowledge graphs TKG research — no production DB yet

Knowledge graph edges and nodes carry timestamps and validity intervals. Facts expire, supersede each other, and have known effective dates. "Vendor X is ISO-certified" is not a timeless fact — it is true from t₁ to t₂. Retrieval follows temporal validity, not just structural reachability.

Why it matters here: PAAP's temporal_decay is a manual approximation of what a temporal knowledge graph would give natively. A TKG would make certificate expiry, contract renewal dates, and regulatory update dates first-class retrieval constraints — not just a decay penalty applied after the fact.

No production database implements full TKG semantics yet. Closest: Wikidata's temporal properties, property graphs with valid_from/valid_to edge attributes. This is the most natural evolution path for the AuthorityEdge model in section 4.

Where this work sits on the research map: The vector + graph + semantic stack is the production ceiling. This project builds a governance layer on top of it — DSC + PAAP + data gateway — that most production agent systems omit entirely. In research terms, this is a deployed approximation of a neurosymbolic retrieval system: symbolic scope contracts enforce completeness, neural retrieval handles semantic variation, and a deterministic authority formula (inspired by PageRank) handles quality ranking. The gap that no existing system — research or production — has closed is a database where these three concerns (completeness, authority, semantic recall) are unified as first-class query primitives rather than stitched together in application code. That gap is the contribution.
