2 — Retrieval Architecture: Top-k vs Complete Coverage

Standard RAG uses top-k vector similarity as the sole retrieval mechanism. In enterprise decision systems this is structurally insufficient — not just "less accurate," but wrong by design for governed decisions where evidence completeness is a compliance requirement.

⚠ Standard RAG — top-k similarity

1 Index all documents as dense vectors (embeddings).
2 Query arrives → embed query → cosine similarity over index.
3 Return top-k most similar chunks as context.
4 Model reasons over top-k and produces an answer.
Enterprise failures:
A compliance blacklist written in legal language ranks low against a business query — silently excluded at position k+1.

You do not know what you do not know. If the critical document is #11 and k=10, the decision is made on incomplete evidence with no warning.

"Similar" ≠ "relevant." A vendor contract clause uses different terminology than the procurement query. Vector distance is not legal relevance.

No completeness guarantee. The model cannot tell you "I did not see the ISO 27001 certificate" — it was never retrieved.

✓ DSC + PAAP — complete coverage with authority ranking

1 DSC defines required evidence classes for this decision type (e.g. procurement must include: vendor_proposal, compliance_rule, signed_contract).
2 Data gateway retrieves all documents in each required class — no k cutoff within a required class.
3 PAAP scores each document by authority_weight × temporal_decay × corroboration — not by similarity to the query.
4 Worker receives a complete, scored, ranked evidence bundle. Missing required class = validation failure before the worker even runs.
Enterprise guarantees:
Compliance evidence is always present — DSC marks compliance_rule as required, not optional.

The model knows what it did not find. An empty required class is an explicit signal, not a silent omission.

Authority ranking (not similarity ranking) surfaces signed contracts above analyst estimates regardless of terminology match.

Complete audit trail: every retrieved document, its authority score, and why it was included or excluded is recorded.
The key insight: In enterprise decisions, a missing piece of evidence is worse than having lower-quality evidence in context. A vendor recommendation made without seeing the compliance blacklist is a legal liability — regardless of how semantically "similar" the retrieved documents were to the query. Top-k retrieval optimises for relevance. DSC + PAAP optimises for completeness first, then quality. These are different objectives and the difference matters most when stakes are high.

3 — Authority Scoring: PageRank Principle Applied to Evidence

PAAP evidence scoring is not an arbitrary formula — it implements the same core insight as PageRank: not all sources are equal, and authority is a property of the source network, not of content similarity.

PageRank — web authority propagation

PR(A) = (1−d) + d × Σ PR(Tᵢ) / C(Tᵢ) d = damping factor (0.85) PR(Tᵢ) = rank of page linking to A C(Tᵢ) = total outbound links from Tᵢ

A page is authoritative if authoritative pages link to it. Being linked by many low-authority pages matters less than one link from a high-authority source. Authority propagates through the link graph.

PAAP — evidence authority scoring

score(e) = authority_weight × temporal_decay(age) × corroboration_factor(n) × (1 − conflict_penalty) authority_weight = source tier (signed_contract=1.0, analyst_estimate=0.4) temporal_decay = e^(−λ × age_days) corroboration = 1 + log(n_independent_sources) conflict_penalty = fraction of contradicting sources

Evidence is authoritative if it comes from a trusted source tier, is recent, is corroborated by independent sources, and is not contradicted. No LLM in this path — fully deterministic.

Direct parallels

PageRank: base authority of source page → PAAP: authority_weight per source tier (signed contracts = 1.0, like a top-domain site)
PageRank: inbound link count from authoritative sources → PAAP: corroboration_factor — multiple independent sources citing the same fact raise its score
PageRank: freshness signals in news/real-time search → PAAP: temporal_decay — a 3-year-old price quote has lower authority than last week's
PageRank: spam detection reduces rank of suspicious pages → PAAP: conflict_penalty — a source directly contradicted by higher-authority sources is penalised
PageRank: operates on graph of links, not on page content → PAAP: operates on evidence metadata, not on content similarity — orthogonal to vector search
Why this matters for the thesis: PAAP and vector retrieval are complementary, not competing. Vector search finds potentially relevant evidence. PAAP ranks it by authority once found — and DSC ensures the required classes were searched in the first place. The three layers together implement what PageRank does for web search: complete coverage (crawl all pages) + authority ranking (PageRank score) + scope enforcement (SafeSearch / content policy). The thesis claim is that enterprise decision systems need all three layers, and current RAG architectures provide only one.

4 — Knowledge Retrieval Stack

Five retrieval layers from coarsest to finest. Each handles a different class of question. A production decision system needs all five; current prototype has only one.

V1
Human-curated knowledge vault (Obsidian)

Domain experts write and maintain authoritative knowledge as linked markdown notes. Procurement guidelines, risk frameworks, vendor evaluation rubrics. The human editorial layer — highest authority weight in PAAP. Obsidian's graph view exposes the link structure that feeds the graph DB. Replaces static knowledge/ folder with an editable, versioned, relationship-aware KB.

planned
Semantic knowledge graph (Kuzu / Neo4j)

Typed business objects (Vendor, Contract, Clause, Risk, Policy, Budget) connected by typed relationships (vendor → supplies → product, contract → governs → vendor, risk → mitigated_by → control). Graph traversal follows paths that flat vector search cannot — "find all clauses in contracts with vendors who have an open risk." Kuzu is embedded (no server), Neo4j for team-scale. Nodes are vector-encoded for hybrid lookup.

planned
Dense vector index (pgvector / Qdrant)

Semantic similarity search over document chunks. Handles terminology variation and paraphrasing. Works best when combined with BM25 (hybrid fusion with RRF re-ranking). Not used as the sole retrieval mechanism — DSC required classes are fetched first, then vector search supplements within allowed_classes. Evidence authority (PAAP) re-ranks the result set before workers see it.

V1
BM25 / keyword index (OpenSearch / pg full-text)

Exact and near-exact term matching. Critical for legal/compliance retrieval where specific clause numbers, regulation codes, and technical identifiers must match exactly. BM25 + vector hybrid consistently outperforms pure vector on factual enterprise queries because domain terminology is sparse and precise. PostgreSQL full-text is sufficient for V1; OpenSearch for high-volume cross-run search.

built
Filesystem keyword search (FilesystemMemory — current)

Simple substring search over knowledge/ markdown files and past run outputs. Covers the prototype use case. No embeddings, no index, no graph. Seed for all layers above — the same MemoryProvider interface is implemented by FilesystemMemory today and by VectorMemory / GraphMemory tomorrow.
