L3 — Component: Decision Runtime

Zooms inside the Decision Runtime container. Components are grouped by responsibility zone. The Governance Kernel (blue) is the project's core differentiator — deterministic, no LLM in its path.

Decision Runtime — Python package: decision_agent
Intake & Architecture
Component
Run Service

Orchestrates the full run lifecycle: create, start, build architecture, generate contracts, complete. Central coordinator for all other components.

modules.runs.service
Component
Decision Router

Classifies incoming task text into a decision_type (procurement, software, story…) using keyword matching with LLM fallback.

modules.decisions.router
Component
Architecture Registry

Stores static, pre-defined architecture definitions (e.g. software-scaffold-build/v1). Looked up before attempting dynamic proposal.

modules.architectures.registry
Component
Proposal Builder

Dynamically generates a worker topology for any domain: goal structure classifier → topology builder → domain/software decomposer → package list.

modules.architectures.proposal
Contracts

Contract Generator

Converts an architecture proposal into worker contracts: read_paths, write_paths, allowed_tools, validators, output_schema, DSC scope, evidence profile, DAR action type.

modules.contracts.generator
Component
Contract Validator

Validates architecture and contract integrity before any worker is dispatched. Checks structural constraints, path safety, and required fields.

modules.contracts.validator
Component
Output Validator

Validates worker output after completion: JSON schema check → DSC scope check → PAAP evidence scoring → DAR authorization trigger.

modules.contracts.output_validator
Execution
Component
Dependency Scheduler

Determines which workers are ready by checking dependency graph status and phase gate state. Returns ready worker IDs; detects deadlocks.

modules.runs.scheduler
Component
Worker Runner

Executes a bounded worker: builds system prompt from contract, runs tool loop, applies JSON-nudge recovery, enforces max_steps, checkpoints after each tool call, retries with exponential backoff.

modules.workers.runner
Component
Tool Provider

Implements all worker tools: read_file, write_file, list_files, run_tests, query_sql (DSC-gated), memory_search, web_search (stub). Enforces read/write path constraints from contract.

modules.workers.tools
Component
Benchmark Runner

Runs evaluation sweeps across PAAP threshold configurations. Auto-approves gates in benchmark mode. Produces CSV + summary.json + claim_check results.

modules.evaluation.runner
Governance Kernel ★ deterministic — no LLM
Component · Core
DSC — Decision Scope Contract

Enforces evidence scope: allowed evidence classes, required classes, out-of-scope markers, phrase blocklist. Applied at output validation and injected into worker system prompts.

modules.governance.dsc
Component · Core
PAAP — Evidence Authority Protocol

Scores evidence records: authority_weight × temporal_decay × conflict_penalty × corroboration_bonus. Deterministic formula. Persists scored EvidenceRecord objects.

modules.governance.paap + paap_score
Component · Core
DAR — Decision Authorization Receipt

Maps proposed action to consequence class (INTERNAL_REVERSIBLE, EXTERNAL_VISIBLE…). Returns ALLOW / DENY / ESCALATE deterministically. Persists AuthorizationReceipt.

modules.governance.dar
Component · Core
LayerConfig

Per-run toggles: dsc_enabled, paap_enabled, dar_enabled, human_gate_enabled. Stored in run-record.json and threaded through every governance call.

modules.governance.layer_config
Infrastructure Seams
Component
Audit Logger

Single append_audit_event() function. Writes typed events to audit.jsonl. Used by every module — the shared event backbone of the entire system.

shared.audit_log
Component
Memory Provider

Implements MemoryProvider interface. FilesystemMemory performs keyword search over knowledge/ and data/memory/. Returns MemoryHit list with scope enforcement.

shared.memory.filesystem
Component
Provider Registry

Resolves the active LLM provider by name. Manages FallbackProvider chain. Workers call complete_with_tools() — never import anthropic or ollama directly.

shared.providers.registry
Component
Run State Deriver

Derives the full run state (worker statuses, gate state, cost totals, phase) by replaying events from audit.jsonl. No mutable state store — audit is the source of truth.

modules.runs.state

Key component flows

FlowComponents in sequence
Task intake Run Service → Decision Router → Architecture Registry (or Proposal Builder) → Contract Generator → Contract Validator
Worker execution Dependency Scheduler → Worker Runner → Tool Provider → (SQLite / filesystem / knowledge base)
Governance chain Worker Runner output → Output Validator → DSC → PAAP → DAR → Audit Logger
Every action All modules → LayerConfig.from_dict() to check whether each governance layer is enabled for this run
Model calls Worker Runner → Provider Registry → (Anthropic provider / Ollama provider / Mock) → Retry wrapper → FallbackProvider
Run state read HTTP Server → Run State Deriver → audit.jsonl replay → derived worker statuses, costs, phase

State Machines

Both state machines are derived by replaying audit.jsonl — the functions derive_run_status() and derive_worker_statuses() in modules.runs.state iterate the event list in order and apply the transition tables below. The audit log is the state machine. No separate state store exists.

Run Lifecycle

Seven derived run statuses. The architecture proposal sub-states (building, approved, contracts_generated) happen inside the created window before ready — they are audit events but do not change the top-level run status until run_ready fires.

    state "created" as created {
        [*] --> static_arch : static architecture\nlookup succeeds
        [*] --> building : architecture_build_started\n[dynamic run]
        state building {
            [*] --> proposing
            proposing --> arch_validated : architecture_proposal_validated
            arch_validated --> arch_approved : architecture_approved
            arch_validated --> arch_rejected : architecture_rejected

    waiting_human --> running : human_answered

    validating --> completed : gate_approved\nor run_completed
    running --> completed : run_completed\n[no gate required]

    validating --> failed : gate_rejected\nor run_failed
    running --> failed : run_failed

    failed --> [*]

Worker Lifecycle

Nine derived worker statuses per worker. Multiple workers run concurrently; each has an independent status. The scheduler reads all worker statuses to decide what to dispatch next — it never holds mutable state. 

stateDiagram-v2
    direction LR

    [*] --> planned : contract_created\nor generated_contract_created

    planned --> assigned : worker_assigned\n(scheduler picks up)
    assigned --> working : worker_started\n(tool loop begins)

    working --> working : worker_heartbeat\n(tool call in progress)
    working --> working : worker_retry_started\n(backoff + retry\nfrom checkpoint)

    working --> needs_human : worker_needs_human\n(model asks operator)
    needs_human --> working : human_answered\n(answer injected)

    working --> submitted : worker_submitted\n(final JSON output produced)

    submitted --> validated : validation_passed\n(DSC ✓ PAAP ✓ DAR ✓)
    submitted --> rejected : validation_failed\n(scope / evidence / schema error)

    working --> failed : worker_failed\n(max retries exhausted\nor unrecoverable error)

    validated --> [*]
    rejected --> [*]
    failed --> [*]

Thesis point — the audit log is the state machine.
derive_run_status(events) and derive_worker_statuses(events) replay audit.jsonl on every read. There is no separate state database. The transition tables above are the complete specification — any implementation that emits the same events in the same order will produce the same state. This makes the system fully auditable, replayable, and testable without mocking state.
