The governance kernel is domain-agnostic. When any decision type arrives, DSC loads the matching scope contract — different required evidence, different authority weights, different consequence rules. The chain that follows (PAAP → DAR → human gate) is identical for all of them.

What changes per domain: the scope contract (required classes, authority weights, consequence table) — configured in schema-map.json and the domain module.   What never changes: the governance kernel itself (DSC enforcement, PAAP formula, DAR policy engine, audit trail). Two weeks of configuration work per new domain. The kernel runs unchanged.

Who assigns authority weights — and how

In PageRank, scores emerge automatically from the link graph — no human assigns them. In PAAP they do not. The base authority weight is a deliberate human decision, assigned once per data source by a named organisational role. PAAP then applies automatic adjustments on top of that base. The two responsibilities are separate and must stay that way.

Manual — assigned once, by a named human role
Source Weight Who decided Why
signed_contract 1.00 Legal counsel Legally binding — highest possible authority
compliance_rule 0.95 DPO / Compliance officer Regulatory requirement — non-negotiable
budget_approval 0.90 CFO / Finance controller Approved by authorised financial officer
vendor_proposal 0.70 Procurement manager Self-reported by vendor — must be independently verified
model_inference 0.00 System architect Model's own reasoning cannot be cited as evidence — blocks hallucination

Stored in schema-map.json. Reviewed when data sources change or organisational policy is updated. One assignment per source type — not per run, not per document.

+
Automatic — computed by PAAP per evidence item, per run
Adjustment How computed Effect
Temporal decay e^(−λ × age_days) A cert expiring next month scores lower than one valid for two years. The older the source, the less it counts.
Corroboration boost 1 + log(n_sources) If three independent sources agree — proposal, certification, ranking all confirm ISO 27001 — the score rises. Like PageRank: agreement from multiple sources amplifies authority.
Conflict penalty × (1 − conflict_fraction) If the proposal claims ISO certified but the compliance database has no record, the score drops. Contradiction between sources reduces authority.

No human input for these — pure arithmetic. Same inputs always produce the same result. This is what makes PAAP deterministic and auditable.

Final score = base weight (human, once) × temporal (automatic) × corroboration (automatic) × (1 − conflict) (automatic)
The human sets the organisational trust hierarchy. PAAP applies physics to it — freshness, agreement, contradiction. Neither can do the other's job. When an auditor asks "why did this source score 1.03?": the answer has two parts — "because Legal assigned compliance_rule a weight of 0.95 in schema-map.json" and "because PAAP computed 0.99 temporal × 1.10 corroboration at runtime." Both parts are inspectable. Both are recorded.

What happens next

The scope contract is injected into every worker's system prompt before it runs. The query_sql tool enforces allowed_tables — if the evaluator tries to query a table not in its contract, the tool returns an error and records a scope violation in audit.jsonl.

After the worker submits, the output validator checks the final JSON: if a worker cites model_inference as a source, it is rejected. If a required_evidence_class is absent from the citations, it is rejected.

📄 scope.json
What DSC cannot do: It cannot check whether the worker actually retrieved data from the required evidence classes — only whether it cited them in output. That is PAAP's job.

What happens next

Scored evidence records are written to evidence/worker_evaluator.json with authority weights, temporal scores, and the overall pass/fail verdict.

If PAAP passes, DAR evaluates the proposed action. If PAAP fails — evidence below threshold — the worker output is rejected and the worker retries from its last checkpoint. The worker never reaches DAR.

📄 evidence/worker_evaluator.json
What PAAP cannot do: It cannot decide whether the proposed action is permitted. That is DAR's job. PAAP only judges the quality of the evidence — not what should be done with it.

What happens next

The receipt is written before any action executes. After the evaluator gets ALLOW, the run continues to the recommender. After the recommender gets ESCALATE, the human gate opens and the run pauses.

The human reviews scope.json, evidence/*.json, authorization/*.json, and the recommendation brief — then approves or rejects. On approval: run_completed.

📄 authorization/worker_evaluator.json 📄 authorization/worker_recommender.json
What DAR cannot do: It cannot judge whether the recommendation is correct or whether the evidence is sufficient — that is DSC's and PAAP's job. DAR only decides whether the action type is permitted given the consequence class and evidence floor.

The chain — one line per step

Task arrives → DSC: define scope → scope.json injected into all worker prompts → Workers run within DSC bounds (query_sql enforces allowed_tables) → Evaluator submits output with evidence_sources declared → PAAP: score each source → evidence/worker_evaluator.json written → PAAP passes → DAR fires: score_vendors → ALLOW → run continues → Recommender submits output → PAAP: score sources → evidence/worker_recommender.json written → PAAP passes → DAR fires: publish_recommendation → ESCALATE → Human gate opens → human reviews all artifacts → approves → run_completed
