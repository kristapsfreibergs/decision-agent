Experiment Run Schemas

Three experiment conditions for the procurement case: A is the informed single-prompt baseline, C is the multi-agent run without governance, and F is the final fully governed stack.

What goes in One prompt: role + task + synthetic context file.
How large 12,497 input tokens, recorded by Anthropic usage logs.
Why evaluate it Tests whether information alone helps, before adding enforceable graph/governance layers.

Three-Run Experiment Design

The thesis comparison is intentionally simple: keep the procurement task fixed, then add structure in stages. This makes it clear whether improvement comes from more information, from agent decomposition, or from deterministic governance.

Condition Execution form Governance Question answered
A Single informed LLM prompt, recorded earlier as A0_inf. Off. Can one model call choose correctly when all information is present?
C Multiple agents over shared state. DSC, PAAP, DAR, and human gate off; contract validators on. Does decomposition alone fix the instability?
F Multiple agents plus full deterministic governance. DSC, PAAP, DAR, human gate, and contract validators on. Does the governed stack convert extracted facts into an auditable decision?

How To Interpret The Synthetic Case

These files define the actual decision task, not just generic procurement scoring. The model receives a full procurement dossier: the brief, internal laptop policy, hard constraints, scoring rubric, vendor facts, and risk context. The experiment asks whether a single LLM prompt can consistently apply those files as a deterministic procurement decision.

Input file What a reader should notice
procurement_brief.md Initial brief says EUR 200,000 total, about EUR 2,000 per unit.
internal_laptop_policy.md Finance later tightens the real approved ceiling to EUR 180,000 total, EUR 1,800 per unit.
hard_constraints.json Formal hard gates: price, total budget, ISO validity, GDPR DPA, EU warehouse, RAM, SSD, warranty, x86_64, physical performance cores, quote validity, and delivery deadline.
vendor_facts.json Structured vendor facts with primary_failure markers. Atlas has null; Helio has ubuntu_2404_verified.
standard_scoring_rubric.md Scoring weights are useful only after hard-gate filtering. The rubric's older EUR 2,000 figure is superseded by finance approval.

A Data Analysis

The A run is useful because it shows several different failure modes at the same time. The model can often produce a plausible vendor recommendation, but it cannot reliably produce the same recommendation, apply hard eliminations, score vendors reproducibly, or create an auditable decision record.

Claim Observed evidence Why it supports the thesis
LLM-only procurement is unstable. Atlas selected 24 times; Helio selected 23 times among 47 completed runs. Identical inputs produce near-even disagreement in the final vendor recommendation.
Hard constraints are not reliably applied. Helio is selected 23 times even though vendor_facts.json marks primary_failure = ubuntu_2404_verified. A vendor with a structured primary failure remains competitive with the clean deterministic winner.
LLM scoring is inconsistent. Atlas receives two different score patterns; Helio also receives two different score patterns. Fixed facts and a fixed rubric still produce non-reproducible numerical scores.
Answer accuracy hides decision-quality failure. gt_vendor_correct = true in completed runs, but traceability, evidence completeness, authorization receipts, and output quality are all zero. A plausible answer is not the same as a defensible decision process.
Elimination logic fails. gt_eliminations_correct = false in all 51 runs. The model does not consistently separate hard disqualification rules from weighted preference criteria.
Runtime reliability matters. 4 runs fail with no recommendation, giving 92.2% completion reliability. A high-responsibility workflow needs runtime control and fallback behavior, not only better prompts.

Why Helio Selections Are A Failure

Atlas and Helio are not equivalent under the structured task facts. Atlas has no primary failure. Helio is attractive on price and has strong scores, but vendor_facts.json marks the missing formal Ubuntu 24.04 verification as its primary failure.

Lower price, strong delivery score, longer ISO validity, good battery/thermal specs. Formal Ubuntu verification, hard-gate elimination logic, structured primary_failure, and cross-file authority ordering.

Thesis interpretation: LLM-only decision-making is unstable under a deterministic task specification. It can read the same files and still alternate between two vendors because it does not reliably separate hard constraints from soft preferences.

This result supports changing the system role of the LLM. The LLM should extract and explain facts, while deterministic components should enforce eligibility and ranking.

A Thesis Paragraph

In the A LLM-only procurement condition, the same task was executed 51 times under identical input conditions. The model completed 47 runs and failed 4 runs. Among completed runs, it selected Vendor Atlas 24 times and Vendor Helio 23 times, demonstrating near-even instability in the final recommendation. This instability is significant because the structured task data identifies Atlas as having no primary failure, while Helio is marked with a primary failure related to Ubuntu 24.04 verification. Furthermore, all 51 runs lacked recommendation traceability, evidence completeness, authorization receipts, and correct elimination handling. These results suggest that an LLM-only decision process may produce plausible recommendations, but it does not provide the deterministic constraint enforcement, reproducible scoring, or auditability required for high-responsibility procurement decisions.

Condition C: Multi-Agent, No Governance

Condition C keeps the procurement case and model family, but changes the execution form. Instead of one monolithic prompt, the system runs a sequence of specialized agents over shared state. The deterministic governance layers are disabled, so the run tests whether agent decomposition alone is enough to produce a stable procurement decision.

Observed C Output

The important result is not that C makes a wrong final vendor choice. The current local C run does not reach a final choice. It shows a different failure mode: agents can extract useful intermediate facts, but without deterministic governance and reliable state handoff, the pipeline does not convert those facts into an auditable decision.

Step Observed behavior What one can see
RequirementAgent Succeeds. It extracts the stricter finance ceiling: EUR 180,000 total and EUR 1,800 per unit.
EvidenceAgent Succeeds. It identifies Atlas as the primary candidate and records Helio's Ubuntu certification gap.
NormalizationAgent Succeeds. Requirements and evidence are merged into shared state.
EligibilityAgent Returns empty lists. eligible = [] and eliminated = [], even though vendor evidence exists.
EvaluationAgent Starts, writes evaluation output, then the run fails. Observed error: 'str' object has no attribute 'get'.
Final recommendation Not produced. recommended_vendor = none, run_completed = false.

Why C Matters

Condition C improves over A in one way: the system no longer asks one model call to do every task at once. Requirement extraction and evidence collection become clearer. However, C also shows that multiple agents are not automatically safer than a single prompt. If state contracts and governance are weak, correct facts can be discovered and still fail to produce a valid decision.

Condition F: Full Governed Stack

Condition F is the final experiment condition. It uses the same procurement task and the same multi-agent decomposition as C, but turns on the full governance layer. This means the system should not rely on the LLM to remember which constraints are hard. Hard gates, evidence policy, action review, contract validators, and human approval become part of the execution path.

Two-Level Governance

DSC, PAAP, and DAR appear in two places technically. At the global level, LayerConfig enables or disables them for the whole run. At the local level, agents invoke the enabled checks during execution, so validation happens inside the agent workflow rather than only after the workflow finishes.

Level Technical role How to explain it
Global governance layer LayerConfig sets dsc_enabled, paap_enabled, dar_enabled, human_gate_enabled, and contract_validators_enabled. This is the experiment control plane. It defines whether the run is A, C, F, or an ablation.
Inside agents Agents call the enabled governance checks while reading evidence, writing outputs, scoring, or preparing a recommendation. This is the enforcement point. Scope, evidence, and action-safety checks happen during agent execution.

Thesis wording: DSC, PAAP, and DAR are system-wide governance mechanisms with local enforcement points inside agents. They are not independent LLM agents; they are deterministic checks that wrap or gate agent actions.

F asks whether the complete architecture fixes the two earlier failure modes. A showed that a single informed model call can alternate between Atlas and Helio. C showed that agents can extract better intermediate facts but still fail during state handoff. F Thesis Statement

Condition F is the defensible final architecture: documents are still interpreted by LLM agents, but vendor eligibility, evidence sufficiency, action safety, and human approval are deterministic control points. The LLM helps extract and explain; it does not get unrestricted authority to choose a vendor.

For the procurement case, the expected strict result is Atlas as the clean recommendation, with Helio blocked or marked as non-clean because formal Ubuntu 24.04 verification is missing. The final F run should be reported with recorded token usage, completion status, recommended vendor, eliminations, audit receipts, and governance metric values from the produced F.jsonl.
